对LSI模块的理解

发表于 2017-10-15

LSI的基本假设

如果两个词多次出现在同一文档中，则这两个词在语义上具有相似性。LSI使用大量的文本构建一个矩阵，该矩阵的一行代表一个词，一列代表一个文档，矩阵元素代表该词在文档中出现的次数。该矩阵为稀疏矩阵。

算法流程

  1. 构建矩阵 W
  2. 对矩阵 W 进行降维，找到 W 的一个低阶相似（降维的原因在下一部分说明）
  3. 对降维后的矩阵 W 做 SVD 分解，得到三个矩阵 U(c×l)、S(l×l)、V(n×l)
  4. 对物品 i 通过 d_i = U(transpose)d_i 映射到隐式空间，d_i 为 W 中第 i 列
  5. 对内容 c 通过 w = =w_cVS 映射到隐式空间，w_c 为 W 中第 c 行
  6. 对用户 u 通过类似第 5. 的方法映射到隐式空间，其中向量的每一项表示这个用户对特定物品的过去评分
  7. 把投影的物品向量和投影的用户向量做余弦相似度计算，使用topN方法取其前n个形成推荐列表

降维的原因

  • 初始矩阵W太大，难以处理
  • 初始矩阵W有噪音，通过降维处理可以达到去噪音的效果
  • 初试矩阵W过于稀疏
  • 经过降维处理后，可以解决部分同义词问题和二义性问题

输入输出

输入1

数据名：LI_text
读取路径：file:///data/ubdata/svd-test/with-directors-and-mainactors/label-item-matrix.coo
格式：Dataset<Row>
内容：矩阵的每行是两个整数和一个浮点数，为了方便在后面构造矩阵，设置浮点数都为1
中间操作：将LI_text转为LICSR，对其进行SVD分解，保留latent_rank个特征向量，再对物品进行投影降维
输出：items_proj，其每行代表一个物品

输入2

数据名：UI_text
读取路径：file:///data/ubdata/svd-test/with-directors-and-mainactors/user-item-train.dat
格式：Dataset<Row>
内容：矩阵的每行是两个整数和一个浮点数，为了方便在后面构造矩阵，设置浮点数都为1
中间操作：将UI_text转为UICOO，对用户进行投影降维
输出：users_proj，其每行代表一个用户

输出

数据名：recomm_list
格式：Dataset<Row>
操作函数：topN(users_proj, items_proj, N)
内容：有三列，分别为相似度、用户id、物品id

使用场景

为了解决传统向量空间模型存在的多义性和同义性问题，需要找到一种模型，能够捕获到单词之间的相关性，LSI正式基于这样的使用场景提出的模型。该模型使用SVD来对单词-文档矩阵进行分解。SVD可以看作是从单词-文档矩阵中发现不相关的索引变量(因子)，将原来的数据映射到语义空间内。在单词-文档矩阵中不相似的两个文档，可能在语义空间内比较相似。
使用场景举例:

  • 发现词与词之间的关系，可用于同义词、歧义词检测
  • 在翻译好的文档上进行训练，可以发现不同语言的相似文档，用于跨语言检索
  • 通过查询映射到语义空间，可进行信息检索
  • 在低维语义空间对文档进行比较，用于文档聚类和文档分类

总的来说，LSI的使用场景是为了计算文本内容的相似度。

缺点

  • SVD只是一种数学变换，并无法对应成现实中的概念，因此新生成的矩阵的解释性比较差
  • 在一篇文档中，LSI模型会忽略词语的先后顺序，但词语的先后顺序对文档内容是有很大影响的
  • LSI有一个改进PLSA效果要好于LSA

许愿墙 3.0 产品总结文档

发表于 2016-01-14

撰写人：黄宇韬

1. 总体简介

许愿墙 3. 0 于 2015 年 11 月 28 日 0：00 上线，至 2015 年 12 月 4 日晚 12：59 分下线。产品上线期间通过与广东工业大学学生会的积极合作，产品得到了不错的推广效果，用户数量级超出预期。但在上线期间，已在线的产品仍有多次修改，修改次数亦超出预期，说明项目的开发、管理上面仍存在许多问题。

2. 详细数据

总数据：

  • 愿望数： 2223 条【女生愿望数：1095 条 男生愿望数：1128 条
  • 女生实物类愿望数： 622 条
  • 女生耗时类愿望数： 473 条
  • 男生实物类愿望数： 333 条
  • 男生耗时类愿望数： 795 条
  • 女生未被领取愿望数： 611 条
  • 女生被领取愿望数： 484 条
  • 女生被完成愿望数： 153 条
  • 男生未被领取愿望数： 591 条
  • 男生被领取愿望数： 531 条
  • 男生被完成愿望数： 118 条
  • 祝福数： 493 条
  • 用户数： 5509 个 【男生：3252 个 女生：2017 个 无性别：240 个
  • 消息数： 7092 条
  • API 请求数： 93 万次

3. 数据分析

  • 男生愿望数多达 1128 条，男生们只利用了 12 月 4 日一天的权力反转时间就许了超过女生 6 天以来的愿望数总和
  • 愿望的最快被摘时间，在发布后 5 秒时间内即被领取，堪称秒接
  • 女生愿望中，实物类略多于耗时类，实物类关键词为：益力多、零食、小型电子产品如耳机鼠标等。耗时类关键词为：唱歌
  • 男生愿望中，耗时类远超过实物类，耗时类关键词为：唱歌
  • 女生许愿的 6 天时间内，女生未被领取愿望数多达 611 条，在男生用户群如此庞大的情况下，可见男生们并不是非常活跃（可能原因为女生愿望中实物类较多导致）
  • 女生的愿望中，带图片的愿望被领取的比例大于未带图的愿望，头像为“靓丽”人像的愿望被领取的比例较大，可见男生们倾向于领取“有图有真相”的愿望
  • 在最后一天的权力反转中，男生许下的大部分是耗时类愿望，如唱歌、微信上道晚安或早安等（你懂得）结合部分用户反馈，不少用户有“一键添加微信好友”的需求
  • 存在这样一种现象，若某用户的首个愿望被领取过，则其之后许下的愿望也往往被领取了，而首个愿望未被领取的，则其之后要么不再许愿望，要么许下的也未被领取，可谓“人生赢家恒赢”
  • 在最后一天的权力反转中，男生愿望的被领取率接近 50%
  • 一个可能的原因–因男生愿望耗时类远超过实物类，实现难度较小，因此男生的愿望被领取率较高
  • 可不严谨地认为每两个许下愿望的男生中就有一个男生的愿望被领取
  • 在最后一天的权力反转中，被完成率（男生确认完成的）接近 25%，可见广工大的女生们真的是相当给力
  • 祝福数共 493 条，可见用户在祝福墙上的活跃程度远不及许愿墙，因运营期间在祝福墙上策划的活动（联合某些组织展开线上活动，如集赞送礼品等）并未展开，因此对于许多用户来说祝福墙的实际意义仅剩下“祝福”
  • 用户分享愿望或祝福的方式受到很大的限制，因未能有效指引用户点击右上角分享到朋友圈（此方法存在一个弊端，不能具体定向到某一特定的愿望或祝福，而只能定向到整个页面），因此许多用户选择截图分享到朋友圈，该方式实际上非常限制许愿墙的有效传播
  • 用户比例中，男生：女生 ≈ 3:2 该比例远远超出预期，在许愿墙主战场为大学城校区的情况（僧多粥少）下，女生用户数竟如此巨大，可猜测女生们对【女生节，女生们都是女神】这一模式非常感兴趣，从而形成了女生之间良好的信息传达
  • 存在部分外校用户
  • 无性别用户多达 240 个，且其中有不少是【宝贵的女生用户】，因为了防止恶意修改性别做了仅读入微信性别一次的设置，不少女生因无简单方便的渠道修改性别而苦恼
  • 消息数共 7092 条，除去系统消息仍有不少聊天消息，可见愿望对象聊天室的功能确实是需要的
  • API 请求数达到了惊人的 93 万次，远远超过去年 2.0 版本的 36 万次，考虑到用户总数的量级，可见产品的用户粘性较高
  • 用户增长最快的时期均为校学生会官方微信订阅号推送许愿墙推广文章后，由此可见与校学生会的合作是保证用户量级的关键，与校学生会的合作非常重要

4. 竞品分析

仍待补充

许愿墙 3.0 （以下简称 3.0）并无直接竞争产品，因此只与 许愿墙 2.0 （以下简称 2.0）作对比。

  1. 3.0 较 2.0 主要区别在 3.0 新增暗恋匹配功能和祝福墙功能，通过用户反馈可知，暗恋匹配是非常令人惊喜的一个功能
  2. 3.0 在 UI 上的用户反馈信息多于 2.0
  3. 3.0 用户数仅用不到 3 天的时间既超过了 2.0 在 7 天时间里的用户数总和
  4. 3.0 的 API 请求数远超 2.0
  5. 3.0 的搜索和筛选功能不易使用

仍待补充

  • 简化搜索与筛选功能

JavaScript 学习笔记之执行上下文、作用域、作用域链以及 call() 和 apply() 的用法

发表于 2015-07-22

执行上下文

定义：

每当控制器到达 JavaScript 可执行代码的时候，控制器就进入了一个执行上下文。
执行上下文是个抽象的概念。

一系列活动的执行上下文从逻辑上形成一个栈。栈底总是全局上下文，栈顶是当前（活动的）执行上下文。当在不同的执行上下文间切换（退出的而进入新的执行上下文）的时候，栈会被修改（通过压栈或者退栈的形式）。

如果将执行上下文的栈以数组的形式来表现。

JsStack = []

每次控制器进入一个函数（哪怕该函数被递归调用或者作为构造器），都会发生压栈的操作。

当一开始初始化代码的时候，先是全局代码压入栈中。

JsStack = [
  globalContext
]

定义一个函数 foo

function foo() {
  return;
}
foo();

一旦控制器进入这个函数，就会有新的元素会被压栈到栈顶。

JsStack = [
  functionContext,
  globalContext
]

每次函数返回，退出当前活动的执行上下文时，JsStack 就会被执行对应的退栈操作。当完成之后 foo() 之后，JsStack 就只剩下一个执行上下文–globalContext，直到整个程序结束。

资料来源：

http://goddyzhao.tumblr.com/post/10020230352/execution-context

作用域

作用域有全局作用域和局部作用域两个概念。

其中，处于局部作用域内可以访问全局作用域的变量，而处于局部作用域外面是不可以访问局部作用域内的变量的。

先来看下下面的一段代码：

var globalData = "This is a global data";//全局变量

function globalFunction() {
    var localData = "This is a locall data"//局部变量
    console.log(globalData);//打印出"This is a global data"，说明可以访问到全局变量
    console.log(localData);//打印出"This is a locall data"
    globalData = "Changed the global data";//修改全局变量
    console.log(globalData);打印出"Changed the global data"，修改全局变量成功
    function innerFunction() {
        var innerData = "This is a inner data";
        console.log(innerData);
        console.log(localData);//打印出"This is a locall data"
        console.log(globalData);//打印出"Changed the global data"，说明可以访问到全局变量
    }
    innerFunction();
    //console.log(innerData);如果取消注释这句话，则会出现错误
}

globalFunction();

注释为实际打印出来的效果，因此我们可以看到：处于全局作用域的变量 globalData 可以在局部作用域内访问，并且可以在局部作用域内修改其值。但如果在 globalFunction() 外部去访问 locallData 这个局部变量的话，则会报错。

透过 innerFunction() 可以进一步理解作用域的概念。当我们在局部作用域 globalFunction() 内访问更深一层次的局部内局部作用域 innerFunction() 里的 innderData 变量时，出现了错误。

作用域链

我们已经知道，在 JavaScript 里一切皆为对象。函数也是对象，拥有一些属性和一系列给 JavaScript 引擎访问的内部属性，其中有一个内部属性是 [[scope]] ，这个内部属性包含了函数被创建时作用域中所有对象的集合，这个集合就是函数的作用域链–[[scope]]，它决定了函数可以访问哪些数据。

当一个函数创建后，它的 [[scope]] 就会填充其可以访问的数据对象。

如下面的函数：

function add(num1,num2) {
    var sum = num1 + num2;
    return sum;
}

当函数 add 刚被创建时， [[scope]] 中会填入一个包含所有全局变量的全局对象。

当函数 add 在执行时，会增加几个变量，在执行函数的过程中会被用到，例如：

var total = add(5,10);

执行函数的时候就会创建前文所提到的执行上下文了，这也是它的一个内部对象。执行上下文定义了函数执行时的环境，其作用域链用来解析标识符（即变量的名字）

这个时候，执行函数 add(5, 10) 的作用域链会被初始化为 [[scope]] 所包含的对象。

其执行上下文中所出现的变量会在函数中按顺序复制到作用域链中，这些变量组成了一个叫做“活动对象(activation object)”的对象

活动对包含了所有局部变量、命名参数、参数集合、this，然后此对象推入作用域链的前端。

当执行上下文销毁后，活动对象也会销毁。

在函数执行过程中，每遇到一个变量，都会经历一次标识符解析过程以决定从哪里获取和存储数据。该过程从作用域链头部，也就是从活动对象开始搜索，查找同名的标识符，如果找到了就使用这个标识符对应的变量，如果没找到继续搜索作用域链中的下一个对象，如果搜索完所有对象都未找到，则认为该标识符未定义。函数执行过程中，每个标识符都要经历这样的搜索过程。

从作用域链看代码优化

我们已经知道了作用域链的结构，在执行一个函数的过程，我们读取一个变量的值，通过标识符去查找，如果标识符的位置很深，那么读取其值的速度就会很慢。因此，查找全局变量是一个很耗费时间的做法，但如果我们需要多次读写某个全局变量时，我们可以采取一个方法：通过一次查找，把它保存在某个局部变量中，再对其进行操作。

call() 和 apply() 方法的补充

定义：

call:调用一个对象的一个方法，以另一个对象替换当前对象。

apply:应用某一对象的一个方法，用另一个对象替换当前对象。

根据定义我们可以知道 call() 和 apply() 都是用来改变引用对象的。

还是通过代码来讲最直接，示例如下，我们首先创建一个对象，它拥有 talk 这个方法

var low = {
    word: 'Hey',
    talk: function(people) {
      console.log(this.word + ',' + people);
      console.log(this === low);
    }
};

我们先直接通过 low 对象来引用这个方法 

low.talk('LaoWen');

浏览器输出：

Hey,LaoWen
true

可以看出当前 this 指向的是引用 talk 方法的 low 对象。

我们另外创建一个对象 fish ：

var fish = {
    word: 'Hi'
};

这个时候通过使用 call 方法来使得 fish 这个对象也能使用 talk 方法

low.talk.call(fish, 'BaoJun');

浏览器输出：

Hi,BaoJun
false

从控制台打印的 false 可以知道 call 方法将引用 talk 方法的对象不是 low 了。call 方法已经将引用对象改成了 fish 。

call 与 apply 的不同

call 方法与 apply 方法的不同就是 call 方法第二个传递的参数值可以是任意的，而 apply 方法第二个传递的参数值必须为数组。

例：

low.talk.call(fish, 'BaoJun');
low.talk.apply(fish, ['BaoJun']);

以上两行代码效果相同。

margin & padding 用法

首先解释一下什么是 margin

根据 w3school 的定义：
margin 简写属性在一个声明中设置所有外边距属性。该属性可以有 1 到 4 个值。

其可能的值有：
auto 浏览器计算外边距。
length 规定以具体单位计的外边距值，比如像素、厘米等。默认值是 0px。
% 规定基于父元素的宽度的百分比的外边距。
inherit 规定应该从父元素继承外边距。

注意事项：
设置 4 个值时遵循 TRBL 规则，即 top right bottom left 上 右 下 左 的瞬时间规则。
设置 2 个值时，第一个值表示 上 下 一致使用的同一个值，第二个值表示 左 右 一致使用的同一个值。比如

margin: 1px 2px; 
margin: 1px 2px 1px 2px;

上面两个是等效的

特殊用法：

margin: 0 auto;

意为上下边距为 0 ，左右边距根据宽度自适应。以上写法可以达到居中的效果。(但要注意给居中的元素一个宽度，并且该元素不能浮动，否则居中无效)

现在来解释下 padding

根据 w3school 的定义：
这个简写属性设置元素所有内边距的宽度，或者设置各边上内边距的宽度。行内非替换元素上设置的内边距不会影响行高计算；因此，如果一个元素既有内边距又有背景，从视觉上看可能会延伸到其他行，有可能还会与其他内容重叠。元素的背景会延伸穿过内边距。不允许指定负边距值。

其值的设定和 margin 类似，但不允许设置为负值。

border 的占位

当给某一元素设置四条等宽的 border 时，值得注意的是，边的起点为没有设置 border 时元素的最左上角那一点。元素的宽度与高度不变，但其位置相比没有设置边时，会被设置的边往右下方挤。

行内元素和块级元素

以下内容来自百度

块级元素具有以下特点：

1.总是在新行上开始，占据一整行；
2.高度，行高以及外边距和内边距都可控制；
3.宽带始终是与浏览器宽度一样，与内容无关；
4.它可以容纳内联元素和其他块元素。

行内元素的特点：

1.和其他元素都在一行上；
2.高，行高及外边距和内边距部分可改变；
3.宽度只与内容有关；
4.行内元素只能容纳文本或者其他行内元素。
不可以设置宽高，其宽度随着内容增加，高度随字体大小而改变，内联元素可以设置外边界，但是外边界不对上下起作用，只能对左右起作用，也可以设置内边界，但是内边界在ie6中不对上下起作用，只能对左右起作用

值得一提的是，块级元素可以通过设置为 display: inline-block; 使其可以在行内容纳其他行内元素。

absolute & relative & fixed 的区别

讲到 absolute 时需要先知道文档流的概念。

简单来说，文档流是文档中可显示对象在排列时所占用的位置。

当给某元素设置 position 为 absolute 时，它会脱离原来的文档流，就好像这个元素不存在了一样，漂浮在其没有设置为 absolute 时的文档位置的地方，其他元素会占据其原来的位置。也就是说 absolute 是相对最近 position 为 relative 或 absolute 的父元素定位，默认相对与 body 定位。

而设置为 relative 时，元素并没有脱离文档流，这个时候给其设置 top 跟 left 值，它只会在原来位置的基础上进行肉眼上的移动，而其在文档流上的位置并没有发生改变。

fixed 将元素直接以浏览器窗口为定位对象，给其设置 top 跟 left 值时，元素将相对固定在浏览器窗口的对应位置上（起点为浏览器窗口的最左上方那一个点）

盒子模型

以下内容来自百科

网页设计中常听的属性名：内容(content)、填充(padding)、边框(border)、边界(margin)， CSS盒子模式都具备这些属性。

这些属性我们可以把它转移到我们日常生活中的盒子（箱子）上来理解，日常生活中所见的盒子也就是能装东西的一种箱子，也具有这些属性，所以叫它盒子模式。
CSS盒子模型就是在网页设计中经常用到的CSS技术所使用的一种思维模型。

每个盒子都有：边界、边框、填充、内容四个属性；
每个属性都包括四个部分：上、右、下、左；这四部分可同时设置，也可分别设置；里的抗震辅料厚度，而边框有大小和颜色之分，我们又可以理解为生活中所见盒子的厚度以及这个盒子是用什么颜色材料做成的，边界就是该盒子与其它东西要保留多大距离。

内容（CONTENT）就是盒子里装的东西；
而填充(PADDING)就是怕盒子里装的东西（贵重的）损坏而添加的泡沫或者其它抗震的辅料；
边框(BORDER)就是盒子本身了；至于边界(MARGIN)则说明盒子摆放的时候的不能全部堆在一起，要留一定空隙保持通风，同时也为了方便取出。在网页设计上，内容常指文字、图片等元素，但是也可以是小盒子（DIV嵌套），与现实生活中盒子不同的是，现实生活中的东西一般不能大于盒子，否则盒子会被撑坏的，而CSS盒子具有弹性，里面的东西大过盒子本身最多把它撑大，但它不会损坏的。
填充只有宽度属性，可以理解为生活中盒子每个HTML标记都可看作一个盒子；

浅谈响应式网页设计

发表于 2015-05-10

从接触前端以来，一直有听到所谓的响应式网页设计，但实际上对其概念也只是一知半解，大概只知道响应式网页设计就是能在各种分辨率下网页的依然能有良好的表现效果。刚好今年在做工作室的招新网站的时候，刚好面临响应式网页设计的需求，借此机会对响应式网页设计有了深一点的理解。如有写得不好或者不对的地方，欢迎留言补充，谢谢指教！

有几种响应式的方法？

###1. 使用 CSS3 的媒体查询
这种方法是使用 CSS3 进行媒体查询，对应不同的分辨率设置不同的 CSS 属性。举个栗子：

@media screen and (max-width: 960px){
    body {
        background: #000;
    }
}

这段代码的意思就是在页面小于 960 px 的时候执行 {} 内的 CSS 代码。

另外值得一提的是，上述代码中添加的 screen 是指当设备在打印页面时使用衬线字体，在屏幕上显示时用无衬线字体。因此，如果所写的页面不考虑打印的情况的话，可以去掉它。

由此我们可以知道当使用这种方法时，我们需要考虑许多种不同分辨率的情况。

所以使用这种方法的优点有：
1.针对性。对不同分辨率的设备有不同的预定义的一套 CSS 代码
2.易用性。这种方法通俗易懂，出现了哪些分辨率的问题可以很快想到解决办法
3.主流。这是目前应用最多的自适应网页设计方案

缺点有：
1.复杂性。需要对不同设备率分辨率编写多套 CSS 代码。越细致的自适应网页可能代码数量越庞大，这会导致不易维护
2.需要多套设计图。对设计师的要求较高，设计师需要对不同分辨率的设备出几套设计图。

###2.固定宽高比
招新网页用的就是这种方法。其大概意思就是，将页面主要内容（背景可占满整个页面）限定在一个固定的宽高比例中，页面内容的元素的定位，大小等全部用百分比的形式来描述，这样就实现了任意分辨率下，页面内容都是等比例缩小或放大的。

限定页面内容的宽高比例用到了 Javascript 去动态的检测设备的分辨率。

现将找新页面检测分辨率的代码抽象，如下:

首先设置页面内容的类名为 animate-container
<div class="animate-container">

Javascript 代码

function fixedPage(x,y) {
  var oHeight = $window.height(),//窗口高度
oWidth = $window.width(),//窗口宽度
      cWidth = oHeight * (x/y),//等比例宽度
      oMargin = (oWidth - cWidth)/2,//左右外间距
      $container = $('.animate-container');
  $container.css({'width':cWidth,'height':oHeight,'left':oMargin});//设置高度为窗口高度，宽度为等比例缩放宽度，并居中显示
}

最后，招新项目中图片的宽高比为 750/1206
所以
fixedPage(750,1206);
